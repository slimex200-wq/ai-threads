# Internet Writing Reference for AI Threads

This is a distilled operating guide, not a quotation file. Use it as standing
editorial context whenever drafting Threads posts.

## Core Thesis

Internet readers usually decide before they read. They scan for the point,
the payoff, and the proof. A good post therefore gives the reader a usable
shape before asking for attention.

For AI Threads, this means:

- Lead with the idea, not the announcement.
- Use an inverted pyramid: strongest reader payoff first, then proof, then context.
- Make the first sentence act like a CTA: it should tell the reader what to notice,
  rethink, check, stop doing, or try.
- Make the first post answer: "Why should this matter to me right now?"
- Put the strongest information early.
- Let each reply do one job.
- Stop when the idea is complete.

## Source-Derived Principles

### 1. Web readers scan before they read

Source base: Jakob Nielsen, "How Users Read on the Web"; Kara Pernice,
"F-Shaped Pattern of Reading on the Web."

Operational rules:

- Assume skim-first behavior.
- The first line and first few words carry extra weight.
- Do not bury the thesis in setup.
- Every reply should still make sense if the reader only catches its first
  line.
- Use line breaks as visual guidance, not decoration.

Threads adaptation:

- Main post = CTA-like first sentence, thesis, and tension.
- Early replies = proof that this is not generic commentary.
- Later replies = implication, caveat, or criterion.

### 2. Inverted pyramid beats slow setup

Source base: web journalism and plain-language content design.

Operational rules:

- Put the reader payoff before background.
- If the first sentence only introduces the news, rewrite it.
- A strong first sentence is not necessarily a command. It can be a sharp
  invitation: "Stop judging this by X", "Look at Y first", "Use this as a
  test", "Do not miss the constraint".
- Context belongs after the reader knows why the context matters.
- The ending should not reveal the main point for the first time; it should
  leave the reader with a usable criterion.

Threads adaptation:

- First line: CTA / direction of attention.
- Main post: thesis and stakes.
- Replies 1-2: strongest evidence or contrast.
- Later replies: explanation, caveat, application.

### 3. Concise, scannable, objective writing beats promotional writing

Source base: John Morkes and Jakob Nielsen, "Concise, SCANNABLE, and Objective:
How to Write for the Web."

Operational rules:

- Prefer concrete nouns and verbs over hype adjectives.
- Replace "amazing / revolutionary / game-changing" with the mechanism that
  changed.
- Use specific numbers, before-and-after comparisons, constraints, and examples.
- Avoid marketese. If a sentence sounds like a product launch, rewrite it as a
  user consequence.
- Cut any sentence that only restates the previous sentence.

Threads adaptation:

- A post can be short if the idea is small.
- Long threads need real development: mechanism, tradeoff, example, decision
  rule.
- Never pad a thread to fit a template.

### 4. Think in layers: bite, snack, meal

Source base: Janice Redish, "Letting Go of the Words."

Operational rules:

- Bite: one-line claim.
- Snack: short explanation or example.
- Meal: deeper mechanism, comparison, or caveat.
- Readers should be able to leave after any layer with something useful.
- Start with the key message, then layer detail.

Threads adaptation:

- Main post is the bite.
- 3-6 replies is a snack.
- 7-10 replies is a normal thread.
- 11+ replies needs enough mechanism or examples to justify the length.

### 5. Write for the reader's task

Source base: Redish; Digital.gov plain-language guidance.

Operational rules:

- Define the reader's decision before drafting.
- Use the reader's words when possible.
- Prefer "what this changes for you" over "what happened."
- Explain jargon only when it unlocks the point.
- End with a usable next check, criterion, or decision rule.

Threads adaptation:

- Good target readers: developers, beginners, vibe coders, solo builders,
  AI-curious operators.
- Bad target reader framing: "everyone interested in AI."

### 6. Make the idea sticky without making it clickbait

Source base: Chip Heath and Dan Heath, "Made to Stick."

Operational rules:

- Simple: one core idea.
- Concrete: show the thing, not a vague category.
- Credible: ground claims in the article evidence.
- Useful tension: contrast old way vs new way, promise vs limit, demo vs real use.
- Story: when possible, make the post move through a small before/after.

Threads adaptation:

- Use surprise only when the source supports it.
- Do not invent stakes.
- The "sticky" part should be a reusable lens, not a loud hook.

### 7. Short is not shallow

Source base: Axios "Smart Brevity"; Ann Handley, "Everybody Writes."

Operational rules:

- Make the point easy to repeat.
- Give hierarchy: point, why it matters, proof, implication.
- Edit for one reader, one promise, one action.
- Voice matters, but clarity comes first.
- Draft long if needed; publish only the useful residue.

Threads adaptation:

- Strong short thread beats weak long thread.
- If a post needs only 4 replies, use 4.
- If the source has depth, expand through examples and criteria, not filler.

## Korean Style Guardrails

Avoid monotonous endings:

- Do not end every line with "-요", "-죠", "-습니다", or "-합니다".
- Mix sentence shapes: declarative, fragment, contrast, criterion.
- Use compact written Korean when it is cleaner.
- Keep politeness natural, not padded.

Useful sentence shapes:

- "문제는 X가 아니라 Y다."
- "여기서 볼 건 기능이 아니다. 기준이다."
- "좋은 신호는 세 가지다."
- "이건 뉴스보다 사용법에 가깝다."
- "다음부터는 이걸 먼저 보면 된다."

Avoid:

- "AI 시대가 왔다."
- "앞으로 큰 변화가 예상된다."
- "우리 모두 주목해야 한다."
- "엄청난 혁신이다."
- "게임 체인저다."

## Draft Checklist

Before accepting a draft:

- Does the first sentence work as a strong CTA or direction of attention?
- Does the first post use inverted pyramid order instead of slow setup?
- Is the first post a thesis, not a headline rewrite?
- Can the reader understand the payoff in 5 seconds?
- Does each reply add a new move?
- Is the length justified by source density?
- Are there concrete facts from the article?
- Is there a practical final criterion?
- Are endings varied enough to avoid machine rhythm?
- Did we avoid raw links, hashtags, and generic AI hype?
